# Student Affairs Domain Context

## Core Language

**Student（学生）**
发起政策咨询或服务事项，并对本人事项拥有知情、补充信息、撤回和结果查看权的高校学生。

**Service Handler（事务经办人）**
在授权部门和事项范围内接收、判断、办理、转派或回访服务事项的工作人员。

**Service Case（服务事项）**
为解决一名学生的一项正式事务诉求而建立的可追踪业务聚合，包含状态、阶段、材料、参与者、待办、决定、结果和时间线。

**Service Definition（事项定义）**
一个版本化的事项办理规则集合，定义所需字段、材料、适用政策、风险规则、处理步骤、部门路由、审批要求和 SLA。

**Policy Version（政策版本）**
由明确发布部门负责，具有发布时间、生效时间、失效时间、适用范围和来源证据的不可变政策快照。

**Work Item（人工待办）**
需要授权经办人执行的明确业务动作，例如补件审核、转派、审批、例外判断或结果确认。

**Conversation Thread（对话线程）**
学生或经办人与 Agent 的交互记录。它可以关联服务事项，但不是正式业务状态。

**Workflow Run（工作流运行）**
一次可持久化、可中断、可恢复的 LangGraph 执行，通过稳定标识关联服务事项、Checkpoint 和任务记录。

**Business Fact（业务事实）**
经过身份校验、规则校验和数据库事务写入的正式记录。模型输出、摘要、向量结果和 Checkpoint 均不能自行成为业务事实。

## Initial Service Journeys

1. 临时困难补助申请：主流程，覆盖信息补全、材料核验、资格预检、人工审批、补件和结果回访。
2. 选课退改政策咨询：政策问答流程，默认不创建服务事项，除非学生明确进入办理。
3. 宿舍报修：辅助流程，覆盖结构化信息收集、风险分级、工单转派和结果确认。

这些场景是首批实现范围，不是写死在系统中的全部业务类型。后续事项通过版本化 `ServiceDefinition` 和必要的专用子图扩展。
